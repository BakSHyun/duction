## 1. 워크로드 분석 (설계는 여기서 출발)

덕션의 트래픽 특성:

| 경로 | 특성 | 위험 |
|---|---|---|
| 조회 (목록·상세·도감) | 읽기 위주, 캐시 가능 | 낮음 |
| **입찰** | 쓰기, **동시성 충돌**, 마감 직전 스파이크 (soft-close로 완화됨) | **높음** |
| 정산 (마감 처리·미결제·승계) | 배치성, 지연 허용 (초 단위) | 중간 |
| 알림 팬아웃 | 쓰기 다량 (작가 팔로워 N명), 지연 허용 | 중간 |
| 이미지 업로드 | 대용량 I/O | 웹 프로세스에서 분리 필수 |

규모 가정: 브라이스 니치 → 동시 접속 수백, 입찰 초당 수십 건이 상한 (인기 분양 마감 순간).
**이 규모에서 초당 수십 건의 트랜잭션은 PostgreSQL 단일 인스턴스가 여유 있게 처리한다.**

## 2. 언어 선택 — "Rust로 갈까?"에 대한 답

### 입찰 엔진의 병목은 어디인가
입찰 처리 = ① 경매 행 잠금 → ② 현재가·프록시 경쟁 계산 → ③ 입찰 기록 쓰기 → ④ 커밋.
②의 "계산"은 정수 비교 몇 번이다. **전체 지연의 95%는 ①·④의 DB 왕복과 락 대기**.
같은 경매에 대한 입찰은 어떤 언어로 짜도 직렬화되어야 한다 (그게 경매의 정합성이다).
→ Rust로 바꿔도 처리량은 거의 그대로고, 잃는 것(풀스택 단일 언어, 타입 공유, 채용 풀, 개발 속도)이 명확하다.

### Rust/Go가 이기는 시점 (Stage 2 트리거)
- 실시간 입찰 **브로드캐스트**가 WebSocket 동시 연결 수만 개 규모가 될 때 (GC 지터·메모리 밀도가 문제)
- 초당 수천 입찰 이벤트 스트림 처리 (시세 집계 파이프라인 등)

### 그때를 위한 준비 (지금 해두는 것)
- 입찰 로직은 이미 `src/lib/bidding.ts` **한 파일에 격리** — 외부 의존은 Prisma뿐
- 이 경계를 유지하면 나중에 `bid-engine` 서비스(언어 무관)로 추출할 때 웹 코드는 함수 호출을 HTTP/gRPC 호출로 바꾸기만 하면 된다
- **금지사항: UI 컴포넌트에서 prisma로 직접 입찰·정산 로직을 짜지 말 것** (경계 침식 방지)

## 3. 서비스 분해 — 장애 도메인 기준

"연쇄적으로 안 터지게"의 핵심은 서비스 개수가 아니라 **뭐가 죽었을 때 뭐가 같이 죽는지**를 설계하는 것.

```
                        ┌─────────────┐
   사용자 ──HTTP──▶     │  web (Next) │──┐
                        │  stateless   │  │ Prisma
                        │  ×N 확장 가능│  │
                        └─────────────┘  ▼
                                   ┌──────────────┐     ┌─────────┐
                        ┌──────▶   │  PostgreSQL  │     │  Redis  │
                        │          │  (진실 원천) │     │ 큐·락·  │
   ┌────────────────┐   │ Prisma   └──────────────┘     │ pubsub  │
   │ worker (정산)  │───┘                               └─────────┘
   │ 10초 폴링 루프 │──────── (Stage 1: BullMQ) ────────────┘
   └────────────────┘
   이미지: R2/S3 (외부) — 웹 프로세스와 완전 분리
```

### 컴포넌트별 역할과 장애 영향

| 컴포넌트 | 역할 | 이것이 죽으면 | 다른 것에 미치는 영향 |
|---|---|---|---|
| **web** | SSR·조회·입찰·등록 | 서비스 중단 (복수 replica로 완화) | 없음 — stateless라 재시작 즉시 복구. 정산은 worker가 계속 돈다 |
| **worker** | 경매 마감 정산, 미결제 처리, 차순위 승계, (미래) 푸시 발송 | 마감 처리 지연 | 웹은 정상. lazy-settle 폴백이 커버. 재시작하면 밀린 것 일괄 처리 (멱등) |
| **PostgreSQL** | 유일한 상태 저장소 | 전면 장애 (단일 장애점) | 관리형 DB(HA·백업)로 해결하는 게 정석. 앱을 쪼개도 이건 안 없어짐 |
| **Redis** | (Stage 1부터) 큐·rate limit | 큐 지연 | **의존 금지 원칙**: Redis 죽어도 입찰·결제는 동작해야 함 (§5-3) |

**웹과 워커를 나누는 이유** (이게 "잘 쪼개기"의 실체):
1. 정산이 폭주해도 (예: 분양 100건 동시 마감) 웹 응답 지연 없음 — CPU·커넥션 풀 분리
2. 웹이 죽거나 배포 중이어도 마감·미결제 페널티는 정확한 시각에 처리
3. 워커는 죽어도 즉시 티 안 남 → 재시작으로 복구 (모든 정산 로직 멱등 설계)

### 왜 더 쪼개지 않는가
입찰·주문·알림을 별도 서비스로 나누면: 분산 트랜잭션 문제(입찰과 알림이 지금은 한 트랜잭션 — 쪼개면 saga/outbox 필요), 네트워크 홉 증가, 장애 조합 폭발. **수백만 유저 전까지는 비용만 있고 이득이 없다.**

## 4. 실행 환경 — Docker Compose

```yaml
services:
  postgres  # 데이터. 볼륨 영속화 + 헬스체크
  redis     # Stage 1부터 소비 (BullMQ·rate limit). 지금은 대기
  web       # Next.js standalone. postgres 헬스 통과 후 기동
  worker    # 같은 이미지, 다른 커맨드. 정산 루프
  migrate   # 일회성: prisma migrate deploy 후 종료 (web/worker가 대기)
```

- **동일 이미지, 다른 진입점**: web과 worker는 하나의 Dockerfile에서 빌드 — 버전 불일치 원천 차단
- 마이그레이션은 앱 기동과 분리된 일회성 잡 — 배포 중 스키마 경합 방지
- 컨테이너별 메모리 상한 → 한 프로세스의 메모리 누수가 호스트 전체를 못 죽임
- 로컬 개발 = 운영과 동일 스택 (dev/prod parity): `docker compose up -d postgres` 후 `npm run dev`

## 5. 연쇄 장애 방지 5원칙 (코드 레벨)

1. **DB가 유일한 조정자** — 입찰 동시성은 `SELECT ... FOR UPDATE` row lock으로 직렬화.
   앱 레벨 분산 락(Redis)을 쓰지 않는 이유: 락 서비스가 또 하나의 장애점이 되고,
   DB 락은 트랜잭션 롤백 시 자동 해제되어 데드락 복구가 공짜다.
2. **정산 멱등성** — worker와 web의 lazy-settle이 동시에 돌아도 안전 (상태 체크 후 전이,
   행 잠금). 두 번 실행 = 한 번 실행과 같은 결과.
3. **바깥 세계는 큐 뒤로** — DB 알림(Notification insert)은 같은 트랜잭션(원자적).
   그러나 외부 발송(웹푸시·알림톡·이메일)은 반드시 큐(BullMQ)로 — 카카오 API가 죽어도
   입찰·결제가 멈추면 안 된다. 실패는 재시도 큐에서 처리.
4. **타임아웃은 기본값이 아니라 명시** — 모든 외부 호출(PG사, 스토리지)에 타임아웃 + 재시도
   상한 + 실패 시 사용자에게 정직한 에러. 무한 대기가 커넥션 풀을 고갈시키는 게
   연쇄 장애의 1번 시나리오다.
5. **리소스 상한 + 헬스체크 + 자동 재시작** — 각 컨테이너 메모리 상한, `/api/health`(DB ping),
   `restart: unless-stopped`. 프로세스는 죽을 수 있다는 전제로 설계하고, 죽으면 빨리 다시 산다.

## 6. 사이드카에 대하여

사이드카는 k8s 파드 안에서 프록시(Envoy 등)를 붙여 mTLS·재시도·서킷브레이커를 앱 밖에서
처리하는 서비스메시 패턴이다. **전제조건이 "서비스가 수십 개"다.** 프로세스 2개인 지금은
같은 목표(장애 절연·관측)를 §5의 원칙 + compose 헬스체크로 달성하는 것이 옳다.
k8s·서비스메시는 Stage 3 (팀과 트래픽이 그걸 요구할 때).

## 7. 단계별 로드맵

| Stage | 시점 | 구성 | 이 단계에서 추가되는 것 |
|---|---|---|---|
| **0 (지금)** | 개발~베타 | compose: web + worker + Postgres + Redis | row lock, 워커 분리, 헬스체크 — **본 문서로 구현 완료** |
| **1** | 정식 오픈 | 관리형 전환: Vercel/Fly(web·worker) + Neon(PG) + Upstash(Redis) + R2 | BullMQ 큐(외부 발송·지연 잡), rate limit, Sentry, 백업 |
| **2** | 동접 수천~ | bid-engine·ws-gateway 추출 (Rust/Go 후보), 읽기 replica, 시세 집계 파이프라인 | 실시간 브로드캐스트 pubsub, 캐시 계층 |
| **3** | 수십만 유저 | k8s + 서비스메시, 서비스 단위 팀 분화 | 사이드카는 여기서 |

**원칙: 단계를 건너뛰지 않는다.** Stage 0에서 Stage 3 아키텍처를 만드는 것이
스타트업이 죽는 대표적인 방법이다. 대신 각 단계에서 다음 단계로 가는 **경계(seam)**를
깨끗하게 유지한다 (§2-3).
