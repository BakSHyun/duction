# 덕션 개발 가이드 — MVP 이후 마일스톤 상세 기획

> 작성일: 2026-07-11
> 상위 기획: [PLANNING.md](./PLANNING.md) · 현재 구현 상태: [README.md](./README.md)
> 원칙: **각 마일스톤은 기획(이 문서) → 구현 → 검증 → 다음 단계** 순서로 진행한다.

---

## 0. 전체 그림

```
[완료] MVP        경매 엔진 · 등록 · 거래 사이클 · 인증
[M1]   탐색       검색 · 필터 · 카테고리 — "원하는 인형을 찾을 수 있다"
[M2]   리텐션     찜 · 알림 — "다시 돌아올 이유가 생긴다"
[M3]   시세       모델 도감 · 낙찰 히스토리 — "여기서만 볼 수 있는 데이터" (해자)
[M4]   배포 준비  Postgres 전환 가이드 · 환경변수 · 체크리스트
─────── 이후 백로그 (§6) ───────
```

우선순위 논리: 유저가 **찾고(M1) → 지켜보고(M2) → 신뢰하고(M3)** 그다음 세상에 공개(M4).
M1~M3은 서로 독립적이지만 이 순서가 유저 여정 순서와 일치한다.

## 1. M1 — 검색·탐색

### 목표
등록 물량이 20건만 넘어도 홈 목록만으로는 못 찾는다. 검색과 필터는 거래 성사율의 전제 조건.

### 기능 명세
- **검색 페이지 `/search`** — 모든 상태가 URL 쿼리스트링에 담겨 공유 가능해야 한다 (트위터 공유가 마케팅 채널이므로 중요).

| 파라미터 | 값 | 기본값 |
|---|---|---|
| `q` | 키워드 (제목+설명 부분일치) | 없음 |
| `cat` | 카테고리 slug — 부모 slug면 하위 전체 포함 (예: `body` → 네오·미디·쁘띠·빈티지) | 전체 |
| `auth` | `GENUINE` / `FACTORY` / `UNKNOWN` | 전체 |
| `grade` | `S` / `A` / `B` / `C` (복수 선택) | 전체 |
| `status` | `live`(기본) / `ended` / `all` | live |
| `sort` | `ending`(마감임박) / `newest` / `bids`(입찰많은) / `price_asc` / `price_desc` | ending |

- **헤더 검색창**: 모든 페이지 상단에서 바로 검색 → `/search?q=`
- **홈 카테고리 숏컷**: 대분류 5개 칩 → `/search?cat=`
- **빈 결과 UI**: "조건에 맞는 경매가 없어요" + 필터 초기화 버튼

### DB 변경
없음 (기존 스키마로 커버).

### 완료 기준
- [ ] 필터 3개 이상 조합해도 정확한 결과
- [ ] URL 복사→새 탭에서 동일 결과 재현
- [ ] 종료 경매 포함 검색 시 낙찰가 표시
- [ ] 결과 0건 UI

### 비범위 (나중에)
페이지네이션(무한스크롤), 가격대 범위 필터, 모델명 기반 검색 자동완성 — 물량 생기면.

## 2. M2 — 찜 + 알림

### 목표
경매는 "마감까지 기다리는" 상품이다. 떠난 유저를 다시 불러오는 장치가 없으면 입찰 경쟁이 발생하지 않는다. **"내가 밀렸다" 알림이 재입찰(=가격 상승)의 핵심 트리거.**

### 기능 명세

**찜 (관심 경매)**
- 경매 상세에 ♥ 토글 버튼 (로그인 필요)
- 마이페이지에 "찜한 경매" 섹션 (진행 중 우선, 마감임박순)
- 찜 수를 경매 상세에 노출 ("12명이 지켜보는 중") — 경쟁 심리 자극

**알림 (인앱 — MVP 단계, 웹푸시·알림톡은 백로그)**

| 타입 | 수신자 | 트리거 시점 |
|---|---|---|
| `OUTBID` 입찰이 밀렸어요 | 이전 최고 입찰자 | placeBid에서 리더 교체 시 |
| `WON` 낙찰됐어요 — 24시간 내 결제 | 낙찰자 | 경매 정산 / 즉구 시 |
| `SOLD` 판매됐어요 | 판매자 | 경매 정산 / 즉구 시 |
| `UNSOLD` 유찰됐어요 | 판매자 | 경매 정산 시 |
| `ORDER_CANCELLED` 미결제 취소 | 판매자·구매자 | 결제 기한 초과 시 |
| `PAID` 결제 완료 — 발송해주세요 | 판매자 | 구매자 결제 시 |
| `SHIPPED` 발송됐어요 | 구매자 | 운송장 등록 시 |
| `WATCHED_ENDING` 찜한 경매 마감 임박 | 찜한 유저 | (백로그 — 스케줄러 필요) |

- 헤더에 벨 아이콘 + 미읽음 수 뱃지 (30초 폴링)
- `/notifications` 페이지: 목록, 클릭 시 해당 경매/주문으로 이동 + 읽음 처리, "모두 읽음" 버튼

### DB 변경
```
Notification (id, userId, type, title, body, link, readAt?, createdAt)
```
알림 생성은 비즈니스 로직(bidding.ts, actions.ts) 안에서 같은 트랜잭션으로 — 알림 유실 방지.

### 완료 기준
- [ ] A가 입찰 → B가 더 높게 입찰 → A에게 OUTBID 알림 생성
- [ ] 경매 마감 → 낙찰자 WON + 판매자 SOLD 동시 생성
- [ ] 미읽음 뱃지 정확, 클릭 시 읽음 전환
- [ ] 엔진 시나리오 테스트에 알림 검증 추가

## 3. M3 — 모델 도감 + 시세

### 목표
"브라이스 ○○ 시세"는 지금 어디서도 못 구하는 데이터다. 낙찰 기록이 쌓일수록 강해지는 **해자이자 SEO 유입 채널**. 거래가 없어도 도감 자체가 콘텐츠가 된다.

### 기능 명세

**도감 목록 `/models`**
- 라인 필터 탭 (네오/미디/쁘띠/빈티지)
- 카드: 모델명, 발매년도, 진행 중 경매 수, 최근 낙찰 평균가 (기록 있을 때만)

**모델 상세 `/models/[id]`**
- 모델 정보 (라인, 발매년도, 정가)
- **시세 요약**: 낙찰 건수, 평균가, 최저~최고가 (커스텀 제외 옵션 — 커스텀은 작가 프리미엄이 붙어 시세를 왜곡하므로 기본 제외, 토글로 포함)
- **낙찰 히스토리 테이블**: 낙찰일 · 낙찰가 · 상태등급 · 정품구분 · 커스텀 여부 (종료 경매에서 도출, 별도 테이블 불필요)
- 이 모델의 진행 중 경매 목록 → 즉시 입찰 유도

**연결점**
- 경매 상세의 모델명 → 도감 링크
- 홈에 "모델 도감" 네비 추가

### DB 변경
없음 — `Auction(ENDED_SOLD) + Item.blytheModelId` 조인으로 도출.
(물량 커지면 집계 테이블/캐시 도입 — 백로그)

### 완료 기준
- [ ] 낙찰 발생 → 해당 모델 히스토리에 즉시 반영
- [ ] 커스텀 제외/포함 토글 시 통계 재계산
- [ ] 도감에서 진행 중 경매로 이동해 입찰 가능

## 4. M4 — 배포 준비

### 목표
실서비스 전환 시 코드 수정 없이 설정만 바꾸면 되는 상태. **실제 배포는 계정(Vercel, DB 호스팅, 도메인) 결정 후 별도 진행.**

### 작업 명세
- `DATABASE_URL` 환경변수화 (`.env` + `.env.example`)
- `DEPLOY.md` 작성:
  1. PostgreSQL 전환 절차 (provider 변경 → 마이그레이션 재생성 → 입찰 트랜잭션에 row lock 추가 지점)
  2. 호스팅 권장 조합: Vercel(앱) + Neon/Supabase(DB) + Cloudflare R2(이미지)
  3. 이미지 업로드 전환 지점 (`actions.ts`의 로컬 저장 → R2)
  4. 경매 마감 처리 전환 (lazy 정산 → Vercel Cron 또는 워커)
  5. 시크릿 목록, 도메인·OG 카드 설정
- 프로덕션 필수 체크리스트 (본인인증, 에스크로 PG 심사, rate limit, 이미지 최적화, 백업)

### 완료 기준
- [ ] 로컬에서 `.env`만으로 DB 경로 변경 가능
- [ ] DEPLOY.md 따라 하면 배포 완료되는 수준의 구체성

## 4.5 M5 — 작가 프로필 + 분양 경매 (GTM 핵심)

### 목표
커스텀 작가는 **공급자이자 인플루언서**다. 작가 1명을 영입하면 팔로워 수백 명이 따라온다 (§13.6).
현재 트위터에서 선착순·DM 추첨으로 진행되는 분양을 "팔로우 → 신규 분양 알림 → 경매 입찰"
루프로 플랫폼에 가져오는 것이 이 마일스톤의 전부다.

### 기능 명세

**작가 등록**
- 일반 유저가 `/artist/setup`에서 작가 프로필 작성 (활동명 소개, SNS 링크) → 즉시 작가 전환
- MVP는 자가 등록. 운영 시 심사·인증 배지로 전환 (백로그)

**작가 프로필 `/artists/[id]`**
- 소개, SNS 링크, 팔로워 수, **팔로우 버튼**
- 진행 중인 경매 (분양 중), 분양 완료 이력 (낙찰가 공개 — 작가 네임밸류의 근거)
- 거래 실적·평점

**작가 디렉토리 `/artists`** — 팔로워순 정렬, 헤더 네비에 "작가" 추가

**팔로우 → 분양 알림 (킬러 루프)**
- 작가 팔로우: `ArtistFollow` 테이블
- 작가가 새 경매 등록 시 → 팔로워 전원에게 `NEW_LISTING` 알림 ("달빛공방 새 분양이 시작됐어요")
- 경매 상세의 판매자 박스: 작가면 프로필 링크 + 팔로우 버튼 노출

**분양 표시** — 작가가 등록한 커스텀 완성품(customArtist=본인)은 카드·상세에 "분양" 배지

### DB 변경
```
User        += isArtist, artistBio, artistSns
ArtistFollow (followerId, artistId, createdAt)  @@id([followerId, artistId])
Notification.type += NEW_LISTING
```

### 완료 기준
- [ ] 작가 등록 → 디렉토리·프로필 노출
- [ ] A가 작가 팔로우 → 작가가 경매 등록 → A에게 NEW_LISTING 알림 (엔진 테스트 포함)
- [ ] 프로필에서 분양 완료 낙찰가 이력 확인 가능

### 비범위
작가 인증 심사, 분양 예약 공지(마감 전 티저), 팔로워 대상 우선 입찰권 — 백로그.

## 4.6 M6 — 어드민 (운영 필수)

### 목표
신고를 받고, 문제 경매를 내리고, 악성 유저를 제재할 수 있어야 오픈할 수 있다.
팩토리 사칭(§13.4)의 "적발 시 영구 정지" 정책도 어드민 없이는 집행 불가.

### 기능 명세
- **신고 접수**: 경매 상세에 "신고" 버튼 — 사유 선택(팩토리 사칭 / 짝퉁·금지품목 / 사기 의심 / 외부거래 유도 / 기타) + 상세 입력
- **어드민 콘솔 `/admin`** (User.isAdmin 계정만):
  - 대시보드: 유저 수, 진행 중 경매, 미처리 신고 수, 최근 주문
  - 신고 목록: 미처리 우선. 각 신고에서 → 대상 경매·판매자로 이동
  - 처리 액션: **기각** / **경매 취소**(입찰 전체 취소 + 판매자에게 알림) / **유저 제재**(경고→7일 정지→영구 정지 단계 지정)
- 제재된 유저는 입찰·등록 차단 (placeBid는 기존 로직에 있음 — 등록에도 추가)

### DB 변경
```
User   += isAdmin
Report (id, reporterId, targetType[AUCTION], targetId, reason, detail?, status[OPEN|RESOLVED|DISMISSED], resolution?, createdAt, resolvedAt?)
Notification.type += AUCTION_CANCELLED | PENALTY
```

### 완료 기준
- [ ] 일반 유저 /admin 접근 차단
- [ ] 신고 → 어드민 목록 노출 → 경매 취소 처리 → 판매자 알림 + 입찰 취소
- [ ] 제재 유저의 입찰·등록 차단 확인

## 4.7 M7 — Reserve Price + 차순위 승계 (고가품 보호)

### 목표
고가 희소품 판매자는 "헐값 낙찰" 공포 때문에 경매를 꺼린다. Reserve가 그 공포를 제거한다.
차순위 승계는 미결제 낙찰 파기 시 판매자 손해를 복구한다.

### 기능 명세
- **Reserve price (비공개 최저 낙찰가)**
  - 등록 시 선택 입력 (시작가보다 높아야 함). 금액은 비공개
  - 상세 페이지에 "최저 낙찰가 설정" 여부 + 현재가 도달/미달 상태만 표시
  - 마감 시 최고가 < reserve → **유찰** (입찰자에게 유찰 알림)
- **차순위 승계**
  - 낙찰자 미결제 취소 시 → 차순위 입찰자에게 승계 주문 자동 생성 + 알림
  - 승계가는 차순위의 마지막 표시 입찰가 (본인이 동의했던 금액)
  - 승계 주문은 **미결제해도 페널티 없음** (`isSecondChance`) — 본인이 응찰한 시점과 상황이 다르므로
  - 승계는 1회만 (3순위 재승계 없음 — 운영 복잡도 방지)

### DB 변경
```
Auction += reservePrice Int?
Order   += isSecondChance Boolean @default(false)
Notification.type += SECOND_CHANCE | RESERVE_NOT_MET
```

### 완료 기준
- [ ] reserve 미달 마감 → 유찰 (엔진 테스트)
- [ ] 미결제 취소 → 차순위 승계 주문 + 알림 (엔진 테스트)
- [ ] 승계 주문 미결제 시 페널티 없음 (엔진 테스트)

## 4.8 M8 — 작가 인증 + 분양 예고

### 목표
작가 신뢰도를 플랫폼이 보증(인증 배지)하고, 분양을 "예고 → 대기 → 오픈" 이벤트로 만들어
입찰 경쟁을 극대화한다 (트위터 분양 공지 문화의 플랫폼 버전).

### 기능 명세
- **작가 인증**: 어드민이 작가 목록에서 인증 부여/해제 → 프로필·경매 상세에 "인증 작가" 배지 (자가 등록은 유지, 인증은 별도)
- **분양 예고 (예약 시작 경매)**
  - 등록 시 "즉시 시작" 또는 "예약 시작"(시작 일시 지정, 최대 7일 후) 선택
  - 예약 경매는 `SCHEDULED` 상태 — 노출은 되지만 **입찰 불가**, "N시간 후 시작" 카운트다운 표시
  - 시작 시각 도달 시 자동 LIVE 전환 (settle 로직에서 처리)
  - 작가 팔로워에게는 등록 즉시 예고 알림 발송 (기존 NEW_LISTING 재사용)
  - 경매 기간은 시작 시각부터 기산

### DB 변경
```
User += artistVerified Boolean @default(false)
```

## 4.9 M9 — 상호 평가 (거래 루프의 완성)

### 목표
"이 판매자 믿어도 되나?"에 데이터로 답한다. 평점·태그가 쌓여야 판매자 배지(백로그)의 기반이 된다.

### 기능 명세
- **평가 시점**: 구매확정(CONFIRMED) 후 구매자·판매자가 서로 1회씩
- **입력**: 별점(1~5) 필수 + 태그(복수 선택: 설명과 일치 / 포장 꼼꼼 / 응답 빠름 / 시간 약속 준수) + 한줄 후기(선택)
- **집계**: 평가 생성 시 대상 유저의 ratingAvg·ratingCount 트랜잭션 갱신 (기존 노출 위치 — 경매 상세 판매자 박스, 프로필 — 자동 반영)
- **노출**: 마이페이지 구매확정 주문에 "평가하기" 인라인 폼, 작가 프로필에 최근 받은 후기 5건
- 수정·삭제 불가 (조작 방지), 주문당 1회 (unique 제약)

### DB 변경
```
Review (id, orderId, reviewerId, targetId, rating 1-5, tags JSON문자열, comment?, createdAt)
  @@unique([orderId, reviewerId])
```

### 완료 기준
- [ ] 구매확정 → 양방향 평가 → 평점 집계 정확 (스크립트 검증)
- [ ] 중복 평가 차단
- [ ] 작가 프로필에 후기 노출

## 4.14 M14 — 덕력 시스템 (신뢰의 단일 수치)

### 목표
평점·거래횟수·페널티·인증이 흩어져 있으면 초보 유저는 판단 못 한다.
**덕력 = 이 사람과 거래해도 되는가를 한 숫자로.** 오리 성장 등급이 브랜드와 결합해
게이미피케이션(등급 올리고 싶은 욕구 → 성실한 거래)까지 만든다.

### 점수 규칙 (투명 공개 — 조작 방지를 위해 전부 서버 이벤트 기반)
| 이벤트 | 덕력 |
|---|---|
| 거래 완료 (구매확정) | 구매자·판매자 각 +50 |
| 받은 평가 ★5 / ★4 / ★3 / ★2 / ★1 | +20 / +10 / 0 / −20 / −40 |
| 경매 등록 | +5 |
| 입찰 참여 | +2 |
| 낙찰 후 미결제 파기 | −100 |
| 운영 제재 (단계당) | −200 |

### 등급 (오리 성장 서사)
| 등급 | 덕력 | 의미 |
|---|---|---|
| 🥚 알 | 0~ | 신규 — 낮은 입찰 한도 안내와 결합 (백로그) |
| 아기오리 | 100~ | 첫 거래들을 무사히 완료 |
| 노랑오리 | 300~ | 커뮤니티의 평균적 신뢰 |
| 청둥오리 | 800~ | 검증된 거래자 |
| 황금오리 | 2000~ | 최상위 — 상위 노출 등 혜택 후보 (백로그) |

### 구현
- `User.duckPower` + `DuckPowerLog`(적립 내역 — 투명성) 트랜잭션 내 갱신
- 노출: 경매 상세 판매자 박스, 작가 프로필·디렉토리, 마이페이지(등급·다음 등급까지 게이지·최근 내역)
- 완료 기준: 입찰·확정·평가·미결제 이벤트별 적립 검증 (엔진 테스트), 로그와 합계 일치

## 4.16 M17 — 앱 전환 준비 (PWA) + 발송 채널 + Redis 실사용

### 앱 로드맵 (3단계)
| 단계 | 방식 | 시점 | 비용 |
|---|---|---|---|
| **1. PWA (본 마일스톤)** | 홈 화면 설치 + 웹푸시. 코드베이스 그대로 | 지금 | 반나절 |
| 2. 하이브리드 (Capacitor) | 웹뷰 래핑 → 앱스토어 배포, FCM/APNs 네이티브 푸시 | 유저가 스토어 검색으로 올 때 | 1~2주 |
| 3. 네이티브 (Expo/RN) | 전용 앱. 서버는 그대로, API 계층만 공용화 | PMF 확정 후 | 1~2달 |

원칙: 서버(경매 엔진·에스크로·알림)는 어떤 단계에서도 재작성하지 않는다.
지금의 API 라우트가 그대로 앱의 백엔드가 된다.

### 이번에 구현
- **PWA**: manifest(이름·아이콘·standalone 모드) + 오리 아이콘 PNG(192/512/maskable)
  → 모바일 브라우저 "홈 화면에 추가" 시 앱처럼 설치·실행
- **발송 채널 추상화**: 알림 발송을 채널 인터페이스로 — webpush(활성),
  SMS·카카오 알림톡(어댑터 자리만, 계약 후 키만 넣으면 활성). 중요 알림(낙찰·승계·제재)만
  유료 채널로 보내는 등급 규칙 포함
- **Redis 실사용 — 레이트 리밋**: 입찰(유저당 30회/분)·로그인(IP당 10회/5분).
  **fail-open**: Redis가 죽어도 입찰은 된다 (ARCHITECTURE §5-3)

### 완료 기준
- [ ] manifest·아이콘 응답 + Lighthouse 설치 가능 판정 요건 충족
- [ ] 발송 파이프라인이 채널 루프로 동작 (기존 웹푸시 테스트 통과 유지)
- [ ] 레이트 리밋 동작 + Redis 중단 시 fail-open (스크립트 검증)
